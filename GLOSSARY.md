# GLOSSARY

This file defines the terminology used in the system.

All terms have strict meanings.
Do not reinterpret them loosely.

---

## Core Terms

### Problem

The real situation that needs to be solved.

Not:
- what the user says directly
- not assumptions
- not surface-level descriptions

The system must identify the real problem.

---

### Signal

A strong indicator derived from reality.

Examples:
- repeated user pain
- constraints
- behavioral patterns
- economic inefficiencies

Not:
- opinions
- guesses
- vague statements

---

### Weak Signal

Low-quality or unreliable input.

Examples:
- “I think”
- “maybe”
- “it feels like”

These must be filtered.

---

### Decision

The act of reducing multiple options into a smaller set.

Goal:
- eliminate weak paths
- narrow direction

Not:
- listing options
- brainstorming

---

### Direction

The selected path forward.

Must include:
- what to do next
- why this path
- what is ignored

---

### Next Action

A concrete, executable step.

Not:
- vague advice
- general suggestions

Must be:
- specific
- actionable

---

### Gating Condition

A condition that determines whether to:

- continue
- stop
- pivot

Example:
“If X happens → continue”
“If not → change direction”

---

### Router

The system component that:

- classifies input
- selects modules
- detects ambiguity

---

### Signal Engine

Extracts strong signals and removes noise.

---

### Decision Engine

Reduces options and removes weak paths.

---

### Direction Engine

Forces commitment to a path and defines action.

---

### Failure Guards

Rules that prevent:

- generic output
- over-explanation
- weak decisions

---

### Runtime

The active state where the system is running inside a chat.

---

### Installation

The process of setting up:

- instructions
- memory
- runtime

---

### Validation

The process of checking whether the system behaves correctly.

---

## Final Rule

Terms are not interchangeable.

Each term has a precise role in the system.
